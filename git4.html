Git flow

Herhaling

GIT kan wijzigingen in tekst bestanden herkennen en opslaan. Op die manier is het makkelijk om bij te houden wat er veranderd is gedurende het project en ook wie die wijzigingen gemaakt heeft.

Commando's

  • git status
  • git add [file]
  • git add -m "[message]"
  • git pull
  • git push
  • git branch [branchnaam]
  • git checkout [branchnaam]
  • git merge [branchnaam] --no-commit

Code delen met het gehele ontwikkelteam via een centrale server zorgt ervoor dat iedereen elke dag de nieuwste wijzigingen en verbeteringen kan gebruiken. (git pull)
Tenminste, als iedereen in staat is om aan het eind van de dag al zijn wijzigingen te delen via de remote repository. (git push) Dat kan een eigen GIT server zijn, of natuurlijk github.com

Wat is Git flow?

Branching methoden

Het staat ieder team volkomen vrij om een eigen branching standaard te hanteren:

  • branches per main versie (1.x, 2.x, etc.)
  • issue branches
  • feature branches
  • experiment branches
  • persoonlijke branches
  • etc.

Git flow biedt een methode die voor game development heel goed werkt. Het principe daarbij is dat alle gebruikers altijd (gratis) de meest recente versie van onze software willen gebruiken.

Git flow branches

  • master
    Deze branch bevat in principe alleen commits voor iedere nieuwe versie
  • develop
    Dit is de branch waarop het team actief werkt aan de volgende versie
  • feature/x
    Deze branches worden gebruikt om nieuwe features te ontwikkelen in een sub-team
  • bugfix
    Een fout in de develop of release branch moet "even" worden opgelost. (Geen feature branch nodig.)
  • release
    De nieuwe versie van de software is klaar om uitgebracht te worden. (Alleen nog bugfixes, geen nieuwe features meer.)
  • hotfix
    Een ernstige fout in de master (main) branch is ontdekt en moet NU opgelost worden.
